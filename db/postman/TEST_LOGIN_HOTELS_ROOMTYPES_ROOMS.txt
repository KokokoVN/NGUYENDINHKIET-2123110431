TEST POSTMAN — Login + Hotels + RoomTypes + Rooms (JSON mẫu)
====================================================================

Base URL:
  http://localhost:5066

Header chung (sau khi login, trừ bước 1):
  Authorization: Bearer <accessToken>
  Content-Type: application/json

--------------------------------------------------------------------
1) LOGIN (không cần token)
--------------------------------------------------------------------
POST http://localhost:5066/api/auth/login

JSON — admin:
{
  "username": "admin",
  "password": "123456"
}

JSON — lễ tân:
{
  "username": "letan",
  "password": "123456"
}

Sau khi gọi: copy giá trị "accessToken" trong response → dán vào Bearer.

Lỗi www-authenticate: Bearer error="invalid_token":
  - Đăng nhập lại lấy token mới (token hết hạn hoặc đổi Jwt:Key trong appsettings).
  - Header chỉ một lần "Bearer": Postman chọn Type = Bearer Token → dán ĐÚNG chuỗi token,
    KHÔNG dán thêm chữ Bearer ở đầu.
  - Swagger: Authorize → dán accessToken (thường không cần gõ Bearer).

7) XÓA MỀM — DELETE (không body) — ADMIN/RECEPTION
--------------------------------------------------------------------
DELETE http://localhost:5066/api/rooms/{id}
  → IsActive = false, StatusCode = OUT_OF_SERVICE

DELETE http://localhost:5066/api/roomtypes/{id}
  → IsActive = false
  → Chỉ được khi KHÔNG còn phòng active nào đang dùng loại phòng này.

DELETE http://localhost:5066/api/hotels/{id}
  → IsActive = false
  → Chỉ được khi KHÔNG còn phòng/loại phòng active thuộc khách sạn đó
  → Nên xóa phòng trước, xóa roomtype trước, rồi mới xóa hotel.

8) KIỂM TRA NHẬT KÝ (dbo.AuditLog) sau POST/PUT/DELETE
--------------------------------------------------------------------
Chạy trong SSMS — các Action ghi khi thao tác Hotels / RoomTypes / Rooms:

  HOTEL_CREATE | HOTEL_UPDATE | HOTEL_SOFT_DELETE
  ROOM_TYPE_CREATE | ROOM_TYPE_UPDATE | ROOM_TYPE_SOFT_DELETE
  ROOM_CREATE | ROOM_UPDATE | ROOM_SOFT_DELETE

SQL:
SELECT TOP 50 AuditId, CreatedAt, ActorUserId, Action, EntityName, EntityId, Reason,
       BeforeJson, AfterJson
FROM dbo.AuditLog
WHERE Action LIKE 'HOTEL_%' OR Action LIKE 'ROOM_%'
ORDER BY AuditId DESC;

10) ĐẶT PHÒNG — Bookings (POST cần ADMIN hoặc RECEPTION)
--------------------------------------------------------------------
GET http://localhost:5066/api/bookings
GET http://localhost:5066/api/bookings?hotelId=1&statusCode=CONFIRMED
GET http://localhost:5066/api/bookings/{id}

POST http://localhost:5066/api/bookings
— Cách 1: khách đã có (lấy customerId từ GET /api/customers)

{
  "roomId": 1,
  "customerId": 1,
  "checkInDate": "2026-05-10",
  "checkOutDate": "2026-05-12",
  "adults": 2,
  "children": 0,
  "ratePerNight": null,
  "specialRequest": null
}

— Cách 2: tạo khách mới + đặt phòng trong một lần (KHÔNG gửi customerId)

{
  "roomId": 1,
  "newCustomer": {
    "customerType": "INDIVIDUAL",
    "fullName": "Lê Văn C",
    "phone": "0933444555",
    "email": "c@email.com"
  },
  "checkInDate": "2026-05-10",
  "checkOutDate": "2026-05-12",
  "adults": 1,
  "children": 0
}

→ ratePerNight: null = lấy theo baseRate của loại phòng; hoặc gửi số cụ thể.
→ Chỉ chặn trùng lịch với đơn CONFIRMED hoặc CHECKED_IN (đã hủy / đã trả phòng không chặn).
→ Phòng OUT_OF_SERVICE / MAINTENANCE không cho đặt.
→ adults + children không vượt capacity của loại phòng.

PUT http://localhost:5066/api/bookings/{id}/check-in
  → Tạo bản ghi dbo.Stay (IN_HOUSE), đơn → CHECKED_IN, phòng → OCCUPIED.
  → Response có thêm stayId (dùng để POST dịch vụ).

PUT http://localhost:5066/api/bookings/{id}/check-out
  → Đóng Stay (CHECKED_OUT, CheckOutAt), đơn → CHECKED_OUT, phòng → DIRTY.
  → Cần đã có Stay IN_HOUSE (đơn check-in bằng bản API này). Đơn CHECKED_IN cũ
    trước khi có Stay phải bổ sung Stay trong DB hoặc check-in lại trên đơn mới.

PUT http://localhost:5066/api/bookings/{id}/cancel    (chỉ khi CONFIRMED)
